# 15 Signals · 融合方向产品需求规格

**版本：** 0.1
**日期：** 2026-05-21
**状态：** 设计讨论稿

> 本文档描述 Direction 1（个人心理疗愈日记）与 Direction 3（心理导师对话）的融合方向。
> 核心理念：**对话即是日记，导师即是镜子。**

---

## 一、产品定位

用户每天不是"填日记"，也不是"做心理测评"——而是与一位思想导师展开一段私人对话。导师通过提问与回应，引导用户触及自己当下真实的内心状态。系统在对话背后静默地检测15个心理信号，并由导师在恰当时机自然地将洞察带回对话中。

**一句话定义：**
> 你每天与一位思想家对话，他不仅回应你今天的感受，也记得你上周的模式。

---

## 二、三位导师

| 导师 | 思想核心 | 适合解读的信号类型 |
|------|---------|----------------|
| **西格蒙德·弗洛伊德** | 潜意识、欲望、防御机制、童年创伤、梦的解析 | 焦虑、压抑、渴望、愤怒、梦境、恐惧 |
| **马克斯·韦伯** | 理性化、职业召唤、祛魅、意义危机、新教伦理 | 工作倦怠、身份认同、控制感、空虚、价值迷失 |
| **卡尔·马克思** | 异化、劳动与价值、阶级意识、社会结构批判 | 疲惫、愤怒、失控感、关系张力、自我价值质疑 |

### 三位导师的隐藏张力
> 弗洛伊德说：问题在你内部
> 韦伯说：问题在意义系统
> 马克思说：问题在外部结构

用户选择不同导师，本质上是在选择**用什么框架理解自己的处境**。这是产品最深的价值主张之一。

## 三、核心体验流程

### 3.1 每日对话流

```
① 进入产品
   └─ 导师画像以 Literary Editorial 风格浮现

② 今日信号浮现（1–3条）
   └─ 基于过去7–14天的滚动趋势
   └─ 以极轻的字体呈现，像导师的观察笔记
   └─ 例："焦虑 · 身份认同 · 渴望"
   └─ 停留约2秒后渐隐，导师开口

③ 导师开场白
   └─ 基于今日激活信号生成
   └─ 若存在历史记忆，自然融入趋势观察
   └─ 例（弗洛伊德）："你上周连续三次提到一种'失控感'。
         今天，你最近感到什么在脱离你的掌控？"

④ 用户回应 → Oracle Transition
   └─ 用户写作（全屏聚焦，无干扰）
   └─ 提交 → 文字溶解（1.3s blur + opacity）
   └─ 呼吸等待点（导师"思考"中）
   └─ 导师回应浮现（1.7s 从模糊到清晰）

⑤ 对话循环
   └─ 轮次不限，用户自主决定何时结束
   └─ 每次用户输入都被系统静默评分（15信号）

⑥ 结束对话
   └─ 用户主动退出，无强制结束
   └─ 系统更新信号趋势数据（不显示给用户）
```

### 3.2 信号呈现规则

| 时机 | 呈现方式 | 内容 |
|------|---------|------|
| 对话开始 | 轻字体浮现，约2秒后渐隐 | 当日1–3个活跃信号标签 |
| 对话中（导师主动） | 融入对话语言，自然提起 | 历史趋势观察，不超过1次/次对话 |
| 对话中（从不） | — | 不显示实时评分、不显示信号强度数值 |

### 3.3 趋势记忆规则

导师在以下条件时主动提起历史趋势：
- 同一信号在连续5天以上对话中出现
- 某信号强度在7天内显著上升（>40%）
- 用户提及的内容与上次高强度信号直接相关

提起方式：**永远以对话语言，永远是问句或观察，绝不是数据报告。**

> ✅ "你上周连续出现了'失控感'，今天这种感觉还在吗？"
> ❌ "系统检测到您的焦虑信号在过去7天上升了63%。"

## 四、界面结构

### 4.1 屏幕列表

| 屏幕 | 描述 |
|------|------|
| **导师选择页** | 三位导师并排，名字 + 一句话思想定位，用户首次使用时选择，之后可切换 |
| **对话主页** | 核心体验，4场景状态机（见下） |
| **导师档案页** | 导师生平、核心著作、思想脉络，可在对话间隙访问 |

### 4.2 对话主页 · 4场景状态机

| 场景 | 内容 | 触发条件 |
|------|------|---------|
| **Welcome** | 导师画像 + 今日信号标签浮现 | 进入产品 |
| **Writing** | 全屏文本输入区，极简，导师上一句话以35%透明度显示于上方 | 用户点击回应 |
| **Thinking** | 居中呼吸点（3个），导师画像缩略图 + 会话计数 | 用户提交输入 |
| **Response** | 导师回应文本浮现，导师署名，"继续对话"按钮延迟2.2s出现 | 思考完成 |

### 4.3 视觉规格

沿用 **Literary Editorial** 设计语言：
- 字体：Playfair Display 900（标题/导师名）· EB Garamond（正文/对话）
- 色调：#F6F3EC（背景）· #1B1B18（墨色文字）
- 装饰：导师木刻风格肖像（SVG）
- 信号标签：小号衬线体，字间距0.15em，细线边框，无填充色
- 整体感：19世纪末欧洲学术期刊

## 五、信号检测机制（后端逻辑）

### 5.1 检测时机
每次用户提交一轮对话输入后，系统静默调用 LLM 对该段文字进行15信号评分。

### 5.2 评分维度
每条输入针对15个信号各给出 0.0–1.0 的强度分值。

### 5.3 趋势计算
- 滚动窗口：7天 / 14天
- 活跃信号定义：过去7天平均分 > 0.4，且上升趋势
- 突出信号（开场呈现）：取趋势斜率最大的前1–3个信号

### 5.4 首次使用（无历史数据）
- 第1次对话：系统基于本次输入实时检测，对话结束后确定当次信号
- 第1–3次对话：不呈现趋势型开场白，导师只基于当日内容开场
- 第4次起：趋势数据积累足够，开始呈现信号浮现 + 历史记忆

## 六、导师内容规格

每位导师需要准备的内容资产：

| 内容 | 描述 | 数量 |
|------|------|------|
| **开场白模板** | 按信号组合（1/2/3个信号）的开场话术，带记忆版本和无记忆版本 | 约60条 |
| **回应风格指南** | 该思想家的语言特征、常用意象、提问方式 | 1份/人 |
| **信号关联映射** | 15信号 × 该思想家的理论对应关系 | 1份/人 |
| **导师档案文案** | 生平简介、核心著作、一句话思想定位 | 1份/人 |
| **木刻风格肖像** | SVG 格式，Literary Editorial 风格 | 1幅/人 |

## 七、Non-Goals（本方向明确不做）

- ❌ 独立的信号报告页 / 趋势图表页
- ❌ 用户可见的信号评分数值
- ❌ 多轮对话历史滚动展示（每次对话聚焦当下）
- ❌ 社交分享功能
- ❌ 用户自选信号或自定义导师
- ❌ 超过3位导师（V1）

## 八、开放问题

| 问题 | 优先级 | 备注 |
|------|--------|------|
| 用户是否可以在同一次对话中切换导师？ | 中 | 建议不允许，每次对话绑定一位 |
| 每日对话是否有推荐时长/轮次？ | 低 | 可通过导师开场引导，不做硬限制 |
| 导师回应是完全 LLM 生成，还是模板+生成混合？ | 高 | 影响内容质量与成本，需技术评估 |

## 九、会话生命周期与主题记忆

> 本章描述"对话何时结束、何时总结、主题如何浮现"——是记忆系统的入口层。

### 9.1 设计原则

1. **不自动结束**：用户不主动选择"结束"，系统永远不结束当日会话。
2. **24 小时连续性**：用户中途离开、关闭 App、锁屏均不影响会话状态；24 小时窗口内重新进入视为续接同一会话。
3. **跨日强制收束**：第二天 8:00（用户本地时区）之前，仍未关闭的会话被系统强制关闭并生成总结。
4. **主题浮现而非定义**：会话进行中由系统持续抽取 2–3 个关键词，作为"不断涌现的主题"，允许更新。

### 9.2 会话状态机

```
                    ┌────────────────────────────────┐
                    │                                │
   [新建 OPEN] ──→ [活跃 ACTIVE] ──用户主动结束──→ [关闭 CLOSED] → 触发总结
        │              ↑    │                              ↑
        │              │    └──8:00 跨日──→ [强制关闭]────┘
        │              │
        └──用户重新进入──┘ (24h 内、未关闭 → 续接同一会话)
```

### 9.3 状态转换规则

| 触发事件 | 状态变化 | 副作用 |
|---------|---------|--------|
| 用户首次打开（或上一会话已 CLOSED） | 新建 session，OPEN → ACTIVE | 启动主题抽取定时任务 |
| 用户中途离开（关 App / 锁屏） | 状态不变（保持 ACTIVE） | 无 |
| 用户 24h 内重新进入 | 续接同一 session | 加载 current_themes 与历史轮次 |
| 用户点击"结束今天的对话" | ACTIVE → CLOSED | 异步触发总结任务 |
| 第二天 8:00 系统定时检查 | ACTIVE → 强制 CLOSED | 异步触发总结任务，下次进入时轻量提示 |

### 9.4 边界处理

| 情况 | 处理 |
|------|------|
| 跨日宽限期 | 8:00 到达后，若用户仍在输入，等待用户 5 分钟无输入再关闭，避免打断 |
| 跨午夜会话归属 | 按 `started_at` 归属——凌晨 1 点开始的会话归前一天，到当日 8:00 强制关闭 |
| 时区 | 8:00 指用户本地时间，`users` 表必须存储 `timezone` 字段 |
| 强制关闭后提示 | 下次进入时导师以对话语言提及，例："昨天那段话，我已经留下了笔记。" |

### 9.5 主题关键词抽取

#### 抽取策略：滑动窗口 + 阈值更新

```
状态：current_themes = [{keyword, confidence, first_seen_turn, last_reinforced_turn}, ...]

每 3 轮对话 或 每 5 分钟（取先到者）触发一次重评：
  1. LLM 读取最近 5 轮对话
  2. 输出候选主题（带 0-1 置信度）
  3. 与 current_themes 比对：
     - 候选与现有主题语义相近 → 提升现有主题 confidence
     - 候选为新主题且 confidence > 0.6 → 加入候选池
     - 现有主题超过 4 轮未被强化 → confidence 衰减
  4. 取 Top 3 作为新的 current_themes
```

#### 为什么不每轮重算

- 每轮重算：主题漂移严重，用户感觉"话题被绑架"
- 阈值更新：真正持续涌现的主题才上榜，符合"不断涌现"的语义

#### 主题 vs 15 信号

| 维度 | 15 信号 | 主题关键词 |
|------|--------|-----------|
| 本体 | 固定 15 类 | 自由文本 |
| 生成方式 | LLM 评分 | LLM 抽取 |
| 用户可见 | 永远不可见 | 可呈现于总结、可由导师在对话中提及 |
| 用途 | 趋势检测、记忆检索权重 | 反映"此刻在谈什么" |

### 9.6 会话总结的内容结构

总结在 session 状态变为 CLOSED 时异步生成，结构化存储：

```json
{
  "narrative": "今天你主要在谈论换工作后的不适感……",
  "themes_final": ["职业倦怠", "身份认同", "母亲期待"],
  "signal_snapshot": { "焦虑": 0.72, "失控感": 0.65 },
  "salient_quotes": [
    { "turn_id": 17, "text": "我感觉每天都在演戏" }
  ],
  "open_threads": [
    "提到了和母亲的争吵但没展开"
  ]
}
```

| 字段 | 用途 |
|------|------|
| `narrative` | 1–3 句叙述，可呈现给用户 |
| `themes_final` | 当日最终主题，写入用户语义画像的候选池 |
| `signal_snapshot` | 当日信号高位，写入 signal_trajectory |
| `salient_quotes` | 用户原话摘录，支撑未来"你上周说……"类型的回忆 |
| `open_threads` | 未展开的话题，下次由导师自然拾回 |

### 9.7 Schema 增量

在 `sessions` 表上新增字段：

```sql
sessions
├── id
├── user_id
├── mentor_id
├── started_at              timestamptz
├── last_active_at          timestamptz
├── status                  enum('active', 'closed_by_user', 'closed_by_rollover')
├── closed_at               timestamptz       -- nullable
├── summary                 jsonb             -- nullable, 见 9.6
├── summary_generated_at    timestamptz       -- nullable
└── current_themes          jsonb             -- 主题状态，频繁更新，整体读写
```

### 9.8 后台任务

| 任务 | 触发条件 | 动作 |
|------|---------|------|
| **主题重评** | 每 3 轮对话后（异步） | 更新 `current_themes` |
| **总结生成** | session 状态变为 CLOSED 时 | 生成 summary，写入 episodic memory 与 signal trajectory |
| **跨日强制关闭** | 每小时扫描一次 | 找出"当前时间在该用户时区是 8:00 区间"的所有 active 会话，走总结流程 |

跨时区调度小技巧：不为每个用户单独设 cron，**每小时跑一次全表扫描**，按用户时区判断是否到 8:00。

### 9.9 开放问题

| 问题 | 优先级 | 备注 |
|------|--------|------|
| 强制关闭后的"轻量提示"具体形态 | 中 | 是导师开场白的一部分，还是单独 UI？ |
| 主题关键词是否在会话中可见 | 中 | 可见 → 用户可能"配合"主题；不可见 → 失去仪式感 |
| 用户是否可以"删除"某次会话的总结 | 中 | 涉及记忆系统的可编辑性，与 GDPR 相关 |

## 十、"结束对话"的形态设计

> 这是 V1 中**唯一不可避免的功能性按钮**。一切都要服从一个原则：**结束应像合上一本书，而不是提交一份表单。**

### 10.1 设计原则

1. **永远可达，永远不显眼**：用户在任何时刻都能找到入口，但它不与对话竞争注意力。
2. **结束是一次仪式**，不是一次点击——需要 2.5–3 秒的过渡序列，让"今天到这里"成为一种感觉。
3. **导师可以提议，用户决定**：在合适的时机（高强度释放后、长时间停顿后），导师的回应可以**邀请**用户结束；但**结束动作永远由用户执行**。
4. **无确认对话框**：弹窗式"你确定要结束吗？"会立刻撕毁文学感。改用单步确认（见 10.3）。

### 10.2 入口位置

| 入口 | 形态 | 触发条件 |
|------|------|---------|
| **主入口** | Writing 场景底部，居中，一行极小衬线字：`今日已尽` | 始终可见，但透明度 35%，与导师上一句话同等弱化 |
| **次入口** | Response 场景中，导师署名下方，"继续对话"按钮旁，并列：`今日已尽` | 仅在 Response 场景出现 |
| **导师提议** | 在导师回应的末尾，作为对话的一部分，例："我们今天到这里如何？" 下方出现两个轻按钮 | 见 10.4 触发条件 |

**不放在的地方**：顶部导航、右上角"X"、汉堡菜单——这些位置带强烈的"退出"语义，与意图不符。

### 10.3 交互序列（约 3 秒）

```
用户点击"今日已尽"
   ↓ (即刻)
界面文字与按钮全部淡出至 0% (0.6s)
   ↓
屏幕中央浮现一行：「今天的对话留在这里。」(衬线，35% 透明度，1.0s fade-in)
   ↓ (停留 0.8s)
该行下方浮现：导师签名 + 当日日期 (小号，0.6s fade-in)
   ↓ (停留 1.5s)
整体淡出至导师选择页或下次进入的等待页 (0.8s)
```

**关键**：整个过程**没有任何按钮**。用户点击"今日已尽"之后，结束动作即已完成——后续是仪式，不是交互。

### 10.4 导师提议的触发条件

满足以下任一时，导师回应的尾部追加一行邀请：

| 触发 | 邀请文案示例 |
|------|------------|
| 当前会话 ≥ 8 轮，且最近 3 轮信号均值显著低于会话前段（释放完成感） | "我们今天到这里？" |
| 用户最近一轮表达明确"想停下"（如"我累了"、"今天先这样"） | "好的，那就让我们到这里。" |
| 用户最近输入 ≥ 5 分钟无回应（在 Response 场景） | （不在对话内提，仅让"今日已尽"按钮透明度从 35% 升至 70%） |

**导师邀请的文案规则**：
- 永远是问句或邀请，不是命令
- 风格匹配该导师（弗洛伊德含蓄、韦伯节制、马克思直接）
- 出现频率上限：每个会话最多 1 次

### 10.5 文案系统

`今日已尽` 是默认文案。可针对导师做轻微差异化：

| 导师 | 按钮文案 |
|------|---------|
| 弗洛伊德 | 今日已尽 |
| 韦伯 | 暂歇于此 |
| 马克思 | 今日如此 |

> 这三个词都规避了"结束/提交/退出"的功能性语义，全部是状态描述。

### 10.6 关闭后的下一次进入

下次进入产品时（无论次日早晨还是当晚再开）：

| 情况 | 导师开场处理 |
|------|------------|
| 上次为 `closed_by_user`（用户主动结束） | 开场白可以自然衔接上次：「上一次，你提到……」 |
| 上次为 `closed_by_rollover`（8:00 强制） | 开场白先轻轻致意：「昨天那段话，我留下了笔记。」 然后再过渡到今日 |
| 上次有 `open_threads` | 优先在开场或前两轮中由导师自然提起 |

### 10.7 反模式（明确不要做）

- ❌ "确定要结束今天的对话吗？" 模态弹窗
- ❌ 显示"本次对话共 N 轮 / N 分钟 / N 字"的统计
- ❌ "导师还想继续，你确定要离开吗？" 之类的挽留
- ❌ 提供"暂停"和"结束"两个不同按钮——结束就是结束，暂停就是直接关 App
- ❌ 在结束后弹出评分 / 反馈表

### 10.8 实现备注

- 前端：单按钮组件，点击立刻触发 `POST /session/{id}/close`，**不等待响应**就开始播放仪式动画。
- 后端：写入 `status='closed_by_user'`、`closed_at=now()`，立刻 200 返回，触发异步 `generate_summary` 任务。
- 容错：若 API 失败，仪式动画照常完成；后台带本地队列，下次启动时补发关闭请求。
